CVE-2026-42809 · May 04, 2026 · updated May 05, 2026 · KEV: No

Apache Polaris: Credential Vending with Unvalidated Storage Locations

Apache Polaris can issue broad temporary ("vended") storage credentials during staged table creation before the effective table location has been validated or durably reserved. Those temporary credentials are meant to limit the scope of accessible table data and metadata, but this scope limitation becomes attacker-directed because the attacker can choose a reachable target location. In the confirmed variant, if the caller supplies a custom location during stage create and requests credential vending, Apache Polaris uses that location to construct delegated storage credentials immediately. The stage-create path itself neither runs the normal location validation nor the overlap checks before those credentials are issued.

CVE-2026-42810 · May 04, 2026 · updated May 05, 2026 · KEV: No

Apache Polaris: Wildcard Character Injection in S3 IAM Policies

Apache Polaris accepts literal * characters in namespace and table names. When it later builds temporary S3 access policies for delegated table access, those same characters appear to be reused unescaped in S3 IAM resource patterns and s3:prefix conditions. In S3 IAM policy matching, * is treated as a wildcard rather than as ordinary text. That means temporary credentials issued for one crafted table can match the storage path of a different table. Confirmed testing shows credentials returned for crafted tables such as f*.t1, f*.*, *.*, and foo.* could reach other tables' S3 locations, enabling cross-table data access and modification.

CVE-2026-42811 · May 04, 2026 · updated May 05, 2026 · KEV: No

Apache Polaris: CEL Injection in GCS Credential Access Boundary

Apache Polaris builds Google Cloud Storage downscoped credentials by creating a Credential Access Boundary (CAB) with CEL conditions intended to restrict access to the requested table's storage path. The CEL string is built from the bucket name and the table path derived from namespace and table identifiers. In current code, that path is inserted into the CEL expression without escaping. As a result, a namespace or table identifier containing a single quote and other URI-safe CEL fragments can break out of the intended quoted string and change the meaning of the CEL condition, effectively removing path restrictions and allowing bucket-wide access.

CVE-2025-70067 · May 04, 2026 · updated May 05, 2026 · KEV: No

Assimp: Buffer Overflow in FBX Importer via Unchecked String Copy

Buffer Overflow vulnerability exists in Assimp versions up to 6.0.2 in the FBX Importer. The vulnerability occurs in aiMaterial::AddBinaryProperty, where a property key string from a crafted FBX file is copied into a fixed-size heap buffer using strcpy() without runtime length validation. This allows attackers to execute arbitrary code by supplying malicious FBX files with oversized property keys that overflow the buffer.
